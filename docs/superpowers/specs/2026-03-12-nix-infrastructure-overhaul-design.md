# Nix Infrastructure Overhaul ‚Äî Design Spec

**Date:** 2026-03-12
**Status:** Approved
**Approach:** Full std rewrite (Approach A)

## Overview

Replace readaloud's basic `flake-utils` flake with a proper `divnix/std` cell-based structure. Add comprehensive linting, formatting, flake checks, a Nix package build, and a NixOS service module.

## Flake Inputs

- `nixpkgs` ‚Äî nixpkgs-unstable
- `std` ‚Äî divnix/std (standalone, not following from configs)
- `treefmt-nix` ‚Äî numtide/treefmt-nix

## Cell Organization

Single `app/` cell under `cells/` with block types:

- `devshells "devshells"` ‚Äî development shells (nixago configs inlined here, matching monorepo/land/claude-plugins pattern)
- `installables "packages"` ‚Äî Nix packages (the app build)
- `functions "nixos"` ‚Äî NixOS module definition

`std.growOn` harvests devShells and packages to top-level. The NixOS module is exported **outside of std** in the `growOn` extra outputs attrset (since NixOS modules are system-independent, they cannot go through std's per-system evaluation). Formatter from treefmt-nix. Checks defined as additional flake outputs.

## Dev Shell (`cells/app/devshells.nix`)

### System packages (carried forward from current flake)

- Elixir 1.17, Erlang/OTP 27, Node.js 22
- SQLite, calibre, poppler-utils, inotify-tools

### New tooling

- biome, statix, deadnix, nixfmt-rfc-style, lefthook, conform, eclint

### Environment variables

- `MIX_HOME` and `HEX_HOME` set to local `.mix` and `.hex` directories (matching current flake)
- `MIX_ENV` defaulting to `dev`

### Shell hooks

- Nixago hooks generate/update `treefmt.toml`, `lefthook.yml`, `.conform.yaml`, `.editorconfig`
- `lefthook install` runs in hook.extra
- `mix local.hex --if-missing && mix local.rebar --if-missing` for bootstrapping

## Package Build (`cells/app/packages/readaloud/default.nix`)

### Primary approach: `beamPackages.mixRelease`

1. **Deps fetching** ‚Äî `mixFodDeps` or `mix2nix`-generated deps derivation to pin Hex packages reproducibly
2. **Asset compilation** ‚Äî `preBuild` phase runs `mix assets.deploy`; esbuild and tailwind binaries provided via nixpkgs (override download steps via `ESBUILD_PATH` / `TAILWIND_PATH` env vars)
3. **Release** ‚Äî `mix release readaloud` produces the OTP release
4. **Runtime deps** ‚Äî output wraps the release with: libstdc++, openssl, ncurses, calibre, poppler-utils

### Fallback: Fixed-output derivation

If `mixRelease` proves problematic, fall back to a two-phase fixed-output approach: hash-locked dep fetch, then build.

### Asset pipeline note

Design is asset-pipeline agnostic. If the project migrates to Vite (via `phoenix_vite`), only the asset build phase changes.

### Re-export

`cells/app/packages/default.nix` re-exports the readaloud package as default so `nix build` works. The subdirectory structure (`packages/{default.nix,readaloud/default.nix}`) is a deliberate choice to keep the build definition isolated and extensible.

## Nixago Configs (inlined in `cells/app/devshells.nix`)

Nixago configs are defined inline in `devshells.nix` using `lib.dev.mkNixago`, matching the pattern used in monorepo, land, and claude-plugins (no separate `configs.nix` or registered nixago block type).

**treefmt dual usage:** The treefmt config exists in two places ‚Äî nixago generates `treefmt.toml` for local dev/lefthook use, while `treefmt-nix.lib.evalModule` in `flake.nix` produces the formatter and formatting check for `nix flake check`. Both must configure the same formatters to stay in sync. The `treefmt-nix` evalModule is the source of truth; the nixago treefmt.toml mirrors it for local tooling.

### treefmt.toml

| Language | Formatter |
|----------|-----------|
| Nix (`*.nix`) | nixfmt-rfc-style |
| JS (`*.js`) | biome |
| CSS (`*.css`) | biome |
| Elixir (`*.ex`, `*.exs`) | mix format (via treefmt wrapper) |
| All files | eclint (editorconfig enforcement) |

### lefthook.yml

- `pre-commit`: runs `treefmt --fail-on-change`
- `commit-msg`: runs `conform enforce --commit-msg-file {1}`

### .conform.yaml

- Conventional commits enforced
- Allowed types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ci`, `style`, `perf`
- Scope optional
- Max header length 72

### .editorconfig

- UTF-8, LF line endings, trim trailing whitespace, final newline
- 2-space indent for nix, elixir, js, css

## Flake Checks

All run via `nix flake check`:

| Check | Description |
|-------|-------------|
| `treefmt` | Verifies all code is formatted |
| `statix` | Nix anti-pattern linting |
| `deadnix` | Unused Nix code detection |
| `credo` | Elixir static analysis (`--strict` mode) |
| `dialyzer` | Elixir type checking (may be split out later if PLT build is too slow) |
| `biome-lint` | JS/CSS linting (separate from formatting) |
| `mix-test` | Runs `mix test` for the full umbrella |

Elixir checks (credo, dialyzer, mix test) depend on the `mixFodDeps` derivation for fetched dependencies. These are the riskiest checks to implement ‚Äî running Mix tasks in the Nix sandbox requires pre-fetched deps, proper `MIX_HOME`/`HEX_HOME` configuration, and for dialyzer, a PLT cache strategy. Expect iteration here; dialyzer may need to be deferred if PLT builds in the sandbox prove too slow or complex.

## NixOS Module (`cells/app/nixos.nix`)

The NixOS module is defined in `cells/app/nixos.nix` but exported **outside of std** ‚Äî directly in the `std.growOn` extra outputs attrset as `nixosModules.readaloud`. This is because NixOS modules are system-independent, while std's cell blocks are evaluated per-system. The `nixos.nix` file is simply imported by `flake.nix`.

Provides `services.readaloud`:

### Options

```nix
services.readaloud = {
  enable = true;
  port = 4000;                          # PHX_SERVER port
  host = "readaloud.example.com";       # PHX_HOST
  dataDir = "/var/lib/readaloud";       # DATABASE_PATH base
  localaiUrl = "http://localhost:8080"; # LOCALAI_URL
  secretKeyBaseFile = "/run/secrets/readaloud-secret-key";
};
```

### What it creates

- `systemd.services.readaloud` ‚Äî runs the OTP release
- `ExecStartPre` runs database migrations
- Dedicated `readaloud` user and group
- `StateDirectory = "readaloud"` for data persistence
- Environment variables mapped from options
- `secretKeyBaseFile` read at service start (no secrets in Nix store)
- Runtime deps (calibre, poppler-utils) added to service `PATH`

### Excluded from module

TTS/STT model configuration (`ttsModel`, `ttsVoice`, `sttModel`) is **not** included ‚Äî this belongs in the application layer (model profiles in the DB), not infrastructure.

## Cleanup Task

Remove TTS/STT model env vars (`TTS_MODEL`, `TTS_VOICE`, `STT_MODEL`) from `config/runtime.exs`. These are legacy config that should be managed through model profiles in the application, not environment variables.

## Decisions & Trade-offs

| Decision | Rationale |
|----------|-----------|
| Standalone std (not following configs) | Keeps readaloud self-contained, no cross-repo dependency |
| Single `app/` cell | Single application, multi-cell is overkill |
| mixRelease with fixed-output fallback | Standard Nix approach for Elixir; fallback if native deps cause issues |
| Dialyzer in main checks (for now) | Start strict, split out if PLT build is too slow |
| Biome over eslint+prettier | Faster, single tool for JS/CSS formatting and linting |
| eclint for editorconfig | Actively fixes violations rather than just reporting |
| No TTS/STT model env vars in module | Application-layer concern, not infrastructure |
| Asset-pipeline agnostic build | Possible future migration to Vite |
